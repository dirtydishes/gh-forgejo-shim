gh-forgejo-shim developer docs testing workflow

Testing During Development

Most changes can be tested directly from the checkout. Reinstall the package only when you need to verify the installed gh wrapper or a GUI app such as Codex.app.

The Mental Model

There are three useful testing loops in this project. Pick the smallest one that proves the thing you changed.

Source Loop

Fastest. Run unit tests or invoke the module with PYTHONPATH=src. No reinstall needed.

Installed Wrapper Loop

Use this when testing the real ~/.local/bin/gh wrapper, pipx packaging, entry points, or the installed venv.

Codex.app Loop

Use this when validating GUI behavior. Refresh the installed package, restart the app when needed, and test a fresh action.

Rule of thumb: edit code, run source tests repeatedly, reinstall once when you are ready to test the behavior as Codex.app or another external tool will see it.

Paths Used In Examples

Examples use $HOME so they work for any macOS user. On this machine, $HOME expands to /Users/kell. If your checkout is elsewhere, replace $HOME/dev/gh-forgejo-shim with your repo path.

cd "$HOME/dev/gh-forgejo-shim"

Python Version

The package requires Python 3.11 or newer. If python3 points at Apple's older system Python, tests can fail before reaching the behavior you changed because modules such as tomllib are missing.

python3 --version
/opt/homebrew/bin/python3 --version

Use the newer interpreter explicitly when needed:

PYTHON=/opt/homebrew/bin/python3
$PYTHON -m unittest

Fast Source Testing

Use this loop for normal implementation work. It exercises the files in your checkout, not the package installed into pipx.

python3 -m unittest

Run focused tests while iterating on a specific behavior:

python3 -m unittest tests.test_routing.RoutingTests.test_create_default_output_surfaces_url_for_codex_app
python3 -m unittest tests.test_routing.RoutingTests.test_create_calls_forgejo_client

For direct CLI smoke checks from the checkout, set PYTHONPATH:

PYTHONPATH=src python3 -m gh_forgejo_shim --help
PYTHONPATH=src python3 -m gh_forgejo_shim version

For helper-level checks, import from the source tree directly:

PYTHONPATH=src python3 -c 'from gh_forgejo_shim.routing import format_created_pull_url; print(format_created_pull_url("https://git.example.com/owner/repo/pulls/7"))'

When To Reinstall

You do not need to reinstall after every edit. Reinstall when the thing you are testing runs through the package installed by pipx, such as gh, gfj, gh-forgejo-shim, or Codex.app.

pipx install --force "$HOME/dev/gh-forgejo-shim"

The managed wrapper usually does not need to be rewritten after a reinstall, because it points at the same pipx venv. Reinstall the wrapper only if ~/.local/bin/gh is missing, points somewhere unexpected, or was replaced:

gfj install-shim

Verify the installed path before blaming the code:

command -v gh
head -3 "$(command -v gh)"
gh-forgejo-shim version

A healthy managed wrapper starts like this:

#!/bin/sh
# managed by gh-forgejo-shim
exec /Users/<user>/.local/pipx/venvs/gh-forgejo-shim/bin/python -m gh_forgejo_shim gh "$@"

Editable Install Option

If you are doing repeated installed-wrapper testing, an editable pipx install can make the wrapper pick up most src/ edits without forcing a reinstall every time.

pipx uninstall gh-forgejo-shim
pipx install --editable "$HOME/dev/gh-forgejo-shim"
gfj install-shim

Use a normal forced reinstall when testing packaging metadata, entry point changes, version changes, or anything that should behave exactly like a fresh install.

Testing Codex.app Behavior

Codex.app sees the installed wrapper, not your checkout. For GUI-facing changes, refresh the package, confirm the wrapper, and restart Codex.app if it was already open.

pipx install --force "$HOME/dev/gh-forgejo-shim"
gfj doctor

For PR-link behavior, create a fresh pull request from Codex.app. The default gh pr create output should be shaped so Codex.app can discover a GitHub-style marker while the browser still opens the real Forgejo PR page:

https://forgejo-host/owner/repo/pulls/7#codex-pr=/pull/7

Old completed app runs will not retroactively pick up a new parser or output shape. Test with a new action after refreshing the installed shim.

Capturing Codex Probe Traces

When the app reports unavailable CLI, PR status, or branch data, capture the exact probes first. The shim only writes trace records when FJ_SHIM_TRACE is set.

export FJ_SHIM_TRACE="$PWD/codex-probe-trace.jsonl"
export FJ_SHIM_TRACE_BODY=1
gfj trace smoke
gfj trace summarize "$FJ_SHIM_TRACE"

For branch selector issues, add the temporary git recorder because those probes may bypass gh entirely:

gfj trace git-recorder create "$PWD/codex-probe-trace.jsonl"
# launch Codex with the printed PATH and FJ_SHIM_TRACE values
gfj trace git-recorder remove /path/to/printed/wrapper-dir

The trace includes command argv, cwd, route decision, host/repo, duration, exit code, output sizes, and redacted excerpts when body tracing is enabled. gh auth token output is never copied into excerpts.

Before Commit

A good pre-commit pass for code changes is:

python3 -m unittest
git diff --check

For documentation-only changes, still run a whitespace check on the touched files:

git diff --check -- docs/dev/testing.html

If you changed turn documents under docs/turns/, also follow the repository turn-document checks from AGENTS.md.

Common Failure Modes

  • Source tests pass, Codex.app still behaves old: reinstall the pipx package and restart Codex.app.
  • Terminal uses the wrong gh: check command -v gh and ensure ~/.local/bin is before the real GitHub CLI in PATH.
  • GUI app cannot find gh: run gfj install-gui-path, then restart the GUI app.
  • Forgejo commands route unexpectedly: check configured hosts with gfj config list and run gfj doctor.
  • Need to bypass Forgejo routing once: run the command with FJ_SHIM_HOSTS= for that process.
