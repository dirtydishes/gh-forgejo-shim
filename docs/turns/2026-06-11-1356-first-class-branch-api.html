First-Class Branch API Compatibility

Added GitHub-shaped branch and git-reference API support so Forgejo repositories can satisfy the branch picker and create-branch probes used by GitHub-oriented GUIs.

Scopegh api branch/ref routing, Forgejo branch client methods, docs, and tests.
Issuegh-forgejo-shim-2ey
Validation90 unit tests passed with python3 -m unittest.

Summary

The shim now owns the common GitHub REST branch/ref endpoints when a Forgejo host is allowlisted. Branch lists, branch lookups, matching refs, and create-ref requests return GitHub-shaped JSON instead of falling through to the real GitHub CLI.

Changes Made

  • Added Forgejo client methods for listing branches, fetching one branch, and creating a branch from another branch.
  • Recognized supported gh api branch/ref endpoints in route detection, including --hostname, full API URLs, and {owner}/{repo}/{branch} placeholders.
  • Implemented GitHub-shaped responses for repos/{owner}/{repo}/branches, branches/{branch}, git/ref/heads/{branch}, git/refs/heads, and git/matching-refs/heads/{prefix}.
  • Mapped GitHub's POST repos/{owner}/{repo}/git/refs create-reference shape onto Forgejo branch creation when the requested SHA already matches an existing branch tip.
  • Documented the supported API endpoints and the create-ref limitation in the README.
  • Added unit coverage for API routing, branch list normalization, matching refs, and branch creation from a matching SHA.

Context

The user-visible problem is a branch picker that initially shows only local branches, then fills in after reopening. This also appears in this GitHub-hosted repo, which suggests Codex.app has an async local-Git refresh issue under some conditions. The Forgejo-specific gap was still real: before this change, the shim delegated all gh api calls, so branch API probes in allowlisted Forgejo repos could fail like GitHub calls instead of returning Forgejo-backed branch data.

GitHub documents branch listing under REST API endpoints for branches and reference creation under REST API endpoints for Git references. The implementation mirrors only the branch/ref subset needed for GUI compatibility.

Important Implementation Details

  • Unsupported gh api endpoints still delegate as before. The new route only claims branch and git-reference paths.
  • gh api --hostname git.example.com repos/owner/repo/branches can route without relying on the current checkout, while placeholder endpoints still use the checkout repo.
  • Branch names with slashes are handled by joining the path suffix after branches/ or heads/.
  • Forgejo's branch API creates from a branch name, not an arbitrary SHA. The create-ref shim therefore looks for an existing branch whose tip matches the requested SHA, then creates the new branch from that branch name.

Expected Impact for End-Users

Forgejo repositories should feel more GitHub-native to GUIs that ask gh api for branches or refs. Branch picker data has a direct Forgejo-backed answer, and creating a new remote branch from an existing branch tip can succeed through the GitHub create-reference shape.

  • python3 -m pytest was attempted, but this environment does not have pytest installed.

Issues, Limitations, and Mitigations

  • This does not directly fix Codex.app's local branch-menu refresh delay in normal GitHub repos; that appears to be outside the shim. The mitigation here is to remove the Forgejo-specific API failure path.
  • Create-ref support only works when the requested SHA is already the tip of an existing Forgejo branch. If a GUI tries to create a branch from an unpushed local commit, the user still needs a normal Git push path first.
  • GraphQL branch search is not implemented. If a GUI relies exclusively on gh api graphql for branches, that remains follow-up work.
