Host-Specific Forgejo Auth

Changed the shim so auth-required Forgejo writes fail before making an unauthenticated API request, and made doctor report missing tokens for the exact configured host.

Summary

A failed Codex.app gh pr create against git.dirtydishes.dev showed Forgejo returning HTTP 401 with token is required. The local machine had auth for another configured Forgejo host, so the old doctor check looked healthy while the routed PR create still had no token for the target host.

Changes Made

  • Added an early auth guard for routed write commands: gh pr create, gh pr new, gh pr comment, gh issue create, and gh issue new.
  • Changed gh-forgejo-shim doctor to check auth for each configured Forgejo host instead of only the first host.
  • Added regression tests for the early pr create failure and for mixed per-host auth state.
  • Updated README and configuration docs to explain that auth is host-specific.

Context

The original command was routed correctly to Forgejo, but discover_fj_token("git.dirtydishes.dev") returned no token. Before this change, the shim built a client with None, sent the POST, and surfaced the raw Forgejo 401 response. The live doctor output now reports: found for git.deltaisland.io; missing for git.dirtydishes.dev.

Important Implementation Details

Write-only guard: read paths such as pr list, pr view, and repo view can still run without a token for public repositories. Mutating paths now require host auth before client creation.

Actionable error: missing auth now prints gh-forgejo-shim auth login HOST and gh-forgejo-shim auth import HOST for the routed host.

Doctor semantics: the auth check is healthy only when every configured Forgejo host has a discoverable token. Mixed state becomes a warning that names both found and missing hosts.

+def _auth_check(hosts: tuple[str, ...], env: Mapping[str, str], *, home: Path | None) -> Check:
+    found: list[str] = []
+    missing: list[str] = []
+    for host in hosts:
+        token = discover_fj_token(host, env=env, home=home)

+            found.append(host)
+        else:

+        return Check("auth token", True, f"found for {', '.join(found)}")

Expected Impact for End-Users

Codex.app users should get a direct repair command instead of a raw Forgejo 401 when creating PRs without auth for the target host. Doctor output should also stop hiding mixed auth state when more than one Forgejo host is configured.

Validation

  • Ran PYTHONPATH=src python3 -m unittest tests.test_routing tests.test_doctor: Ran 44 tests, OK.
  • Ran PYTHONPATH=src python3 -m unittest: Ran 104 tests, OK.
  • Ran the original gh pr create shape through the source tree with GH_REPO=git.dirtydishes.dev/dishes/islandflow and confirmed it prints the host-specific auth repair command before any API call.
  • Ran source-tree doctor and confirmed it reports git.dirtydishes.dev as missing auth while preserving the existing token report for git.deltaisland.io.

Issues, Limitations, and Mitigations

  • This does not create a token for git.dirtydishes.dev; it makes the missing auth visible and prevents an unauthenticated write request.
  • Read-only commands are intentionally still allowed without a token for public Forgejo repositories.
  • gh-forgejo-shim doctor may now warn when any allowlisted host lacks auth, even if the current repo uses a different host. The warning names the missing hosts so stale allowlist entries are easy to clean up or authenticate.
