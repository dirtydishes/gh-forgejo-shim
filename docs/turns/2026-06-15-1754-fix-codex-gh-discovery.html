2026-06-15 17:54 Repository implementation Issue gh-forgejo-shim-k6t

Fixed Codex.app GitHub CLI discovery for GitHub and Forgejo checkouts

Codex Desktop was probing remote.upstream.url before showing PR status. The shim now diagnoses and repairs that remote shape, while the installed gh wrapper is harder to break when bootstrap is run from a source checkout.

Summary

Added a first-class Codex remote compatibility check, exposed it in doctor and bootstrap, and added repair-codex-remote to create an upstream alias from origin only when the alias is missing. The live wrapper generation was also hardened so it prefers the installed gh-forgejo-shim command instead of writing a brittle bare Python module invocation.

Changes Made

Codex Remote Probe

Added src/gh_forgejo_shim/codex_remote.py to check and repair remote.upstream.url.

Diagnostics

doctor and bootstrap now report the missing upstream alias and show the exact repair command.

Repair Command

gfj repair-codex-remote adds upstream from origin and copies a distinct origin push URL.

Wrapper Safety

Generated gh wrappers now prefer gh-forgejo-shim gh when the management command is installed.

Context

Codex logs showed repeated requestKind=gh-pr-status failures from git config --get remote.upstream.url across both GitHub repositories and Forgejo repositories. The actual gh command was present and authenticated; the unavailable state came from repository shape, not a missing executable.

Important Implementation Details

  • The repair command does not overwrite an existing upstream remote, so real fork upstreams are preserved.
  • When origin has a separate push URL, the new alias copies it to avoid a fetch-only alias with surprising push behavior.
  • bootstrap still prints repair commands for repo-shape gaps instead of silently changing branch tracking or remote layout.
  • The wrapper still supports the explicit Python-module form for tests and fallback cases, but normal installs now generate a management-command wrapper.

diff --git a/src/gh_forgejo_shim/codex_remote.py b/src/gh_forgejo_shim/codex_remote.py
+def check_codex_upstream_remote(*, cwd: str | None = None) -> CodexRemoteCheck:
+    upstream_url = git_output(["config", "--get", "remote.upstream.url"], cwd=cwd)
+    if upstream_url:
+        return CodexRemoteCheck(True, f"remote.upstream.url points at {upstream_url}")
+
+    origin_url = git_output(["remote", "get-url", "origin"], cwd=cwd)
+    if not origin_url:
+        return CodexRemoteCheck(False, "remote.upstream.url is missing and origin remote is unavailable")
+    return CodexRemoteCheck(
+        False,
+        "remote.upstream.url is missing; Codex.app PR status probes expect it",
+        (f"git remote add upstream {shlex.quote(origin_url)}",),
+    )
+
+def repair_codex_upstream_remote(*, cwd: str | None = None) -> CodexRemoteRepair:
+    code, message = git_run(["remote", "add", "upstream", origin_url], cwd=cwd)
+    if code != 0:
+        return CodexRemoteRepair(False, False, message or "git remote add upstream failed")
+    origin_push_url = git_output(["remote", "get-url", "--push", "origin"], cwd=cwd)
+    if origin_push_url and origin_push_url != origin_url:
+        git_run(["remote", "set-url", "--push", "upstream", origin_push_url], cwd=cwd)

Issues, Limitations, and Mitigations

  • The repair command intentionally does not rewrite an existing upstream remote. If a repository has a real fork upstream that Codex still cannot use, that should be handled manually.
  • This fixes the observed Codex Desktop probe. A future app build could add other repository-shape expectations; doctor is now the place to surface them.
  • While validating, a source-tree bootstrap run briefly rewrote the live wrapper to a brittle Python invocation. The wrapper was repaired immediately and the code now prevents that path when a management command exists.
